# EXP-FIX-04：放大 MoE 网络的尝试与结论（问题 3）

- 日期：2026-06-24　关联代码：`Networks/net_moe.py`、`train_moe.py`（新增 `--out-channel/--depth/--num-heads`）、`mm_infer_check.py`
- 结论（重要的"负结果"）：**放大 MoE（更宽/更多专家/更深）并不改善质量**——训练损失与逐源结构相关性几乎不随容量变化，反而更大的配置更不稳定（塌黑/近常数）。瓶颈**不是容量**，而是 **MoE 与骨干的集成方式**（专家求和导致 FFN 残差幅度被放大、冲淡结构）。下一步应调试 MoE 而非加大它。

## 1. 动机
此前 MoE（mm_moe_v1）虽解除 IR-VIS 塌黑、S0 全过，但**逐源 Y-corr 偏低**（结构没紧跟源）。按"效果不好就把架构弄大/弄深一点"的思路，做一次容量扫描，验证放大是否是解法。

## 2. 实验设置
统一数据默认（on-the-fly、crops_per_task=4000、patch170、任务自适应 max-强度 + aux0.01），6 epoch。扫描容量：

| 配置 | out_channel | 路由专家 | 深度 | 参数量 |
|---|---|---|---|---|
| oc16（基线） | 16 | 4 | 3 | 0.039M |
| oc32 | 32 | 4 | 3 | 0.150M |
| oc64 | 64 | 4 | 3 | 0.588M |
| oc32-e8 | 32 | 8 | 3 | 0.251M |
| oc32-sym（诊断） | 32 | 4 | 3 | 0.150M（**对称损失**，关任务自适应） |

## 3. 结果
**训练损失几乎不随容量变化**（structure 项，越低越好）：

| 配置 | structure | s/epoch | 退化自检 |
|---|---|---|---|
| oc16 | 3.245 | 215 | ALL PASS，但 corrB≈0 |
| oc32 | 3.228 | 251 | **SOME FAIL**（irvis/medical 塌黑），corrB≈0 |
| oc64 | 3.242 | 373（需 bs12，否则 OOM） | ALL PASS，corrB≈0 |
| oc32-e8 | 3.220 | 270 | **SOME FAIL**（irvis 塌黑），corrB≈0 |
| oc32-sym | 3.253(ep5) | 251 | 近常数(std≈0.015)、corrB≈0 |

参照：**稠密网络（同宽 16、对称损失）corrB 高达 0.9+**（EXP-FIX-03 的 p2b）。

要点：
1. **15× 参数（16→64）+ 翻倍专家（4→8），structure 损失只在 3.22–3.25 间抖动**，无单调改善——容量不是限制因素。
2. **逐源结构相关性在所有容量下都≈0**，与稠密网络的 0.9+ 形成鲜明对比。
3. **更大的配置更不稳定**：oc32、oc32-e8 反而把 IR-VIS/医学塌黑/塌成近常数。
4. **oc64 触发 OOM**：当前 MoE 是"所有专家稠密计算再按门控掩码组合"，宽度一大激活显存暴涨（须把 batch 从 32 降到 12）——放大还自带算力/显存惩罚。
5. **诊断（oc32 对称损失）**：去掉任务自适应损失后 MoE 仍 corrB≈0、近常数输出——说明**不只是损失问题**；同样条件下稠密网络却能 corrB 0.9+ → 指向 **MoE 架构集成本身**。

## 4. 根因判断与下一步（不是加大，而是调试 MoE）
- **判断**：稠密 Mlp → MoEFFN 后，每个 token 的 FFN ≈「1 共享专家 + top-2 路由专家」之和（≈3 个 MLP 相加），输出幅度约为单 Mlp 的 3×；残差 `x + mlp(norm(x))` 被放大 → 推过 tanh 易饱和 / 冲淡空间结构 → corrB≈0、近常数。这能解释"与稠密同宽却结构差、且放大无济于事"。
- **下一步（替代"加大"的真正修法）**：
  1. **专家输出归一**：路由专家加权和后除以有效专家数 / 或共享与路由用可学习标量平衡，控制 FFN 残差幅度；
  2. 降低/调度 **aux 权重**（0.01 可能过强、把路由推向均匀、与任务信号打架）；
  3. 路由前/专家后加 **LayerNorm**，或对共享专家与路由专家分别归一；
  4. 复核 **task_router_emb 加性条件** 的尺度（可能压过 token 特征）。
- 这些都是"修 MoE 集成"，而非"把 MoE 弄大"。**本问题的结论即：放大不是解法。**

## 5. 备注
- 容量扫描的 ckpt 为一次性调参产物，已清理（保留 `logs_p3_moe_*.txt` 作证据）。
- 评判仍以 S0 + corr 代理 + ≤6 epoch 为据，最终须 S1 三轴平均排名复核；但"放大无改善"的趋势在多组配置上一致，结论稳健。
